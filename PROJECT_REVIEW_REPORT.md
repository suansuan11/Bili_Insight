# Bili_Insight 项目现状评估与问题整改方案（2026-03-02）

## 1. 需求理解（基于《项目需求》）

当前项目的目标是一个“Java 业务后端 + Python 分析微服务 + 前端可视化”的 B 站舆情与复盘平台，重点能力包括：

- 用户体系（注册/登录/JWT）与角色权限（创作者/品牌方）。
- 面向品牌方的“监测项目管理”（关键词、UP 列表、自动扫描）。
- 分析任务中心（热门榜、单视频复盘、UP 追踪、项目仪表盘）。
- 交互式复盘页（播放器、情绪时间轴联动、ABSA 切面分析与评论筛选）。

> 需求来源：`项目需求.md`。

---

## 2. 当前代码完成度（核查结论）

### 2.1 已具备能力

- Java 后端已提供分析任务主流程 API（提交任务、查状态、评论/弹幕/时间轴查询等）。
- 前端已存在 Dashboard / Analysis / Popular 等页面与路由，不是“空白前端”。
- Python 服务可正常模块化导入，代码语法层面可通过 `compileall`。

### 2.2 与需求相比的核心缺口

目前代码更像“视频分析 MVP”，离“完整 SaaS 舆情平台”仍有明显距离，特别是在**账号体系、权限模型、品牌方项目监测闭环**方面。

---

## 3. 发现的问题与具体解决方案

## P0（需优先处理）

### 问题 1：缺失用户认证与角色权限主链路（与需求不一致）

**证据**
- 需求明确要求 Spring Security + JWT、角色区分与数据隔离。 
- 现有 Java 控制器主要为分析、热门、看板，没有用户认证/鉴权控制器与对应权限边界。 

**影响**
- 任何人可直接调用分析接口；无法支持“品牌方专属功能”；无法实现多租户数据隔离。

**解决方案（可执行）**
1. 新增 `user`、`role`、`user_role`、`monitor_project`、`project_video` 等表。  
2. 引入 Spring Security + JWT 过滤器，完成登录、刷新 token、登出黑名单（可选 Redis）。  
3. 在接口层按角色加注解（如 `@PreAuthorize("hasRole('BRAND')")`）。  
4. 所有任务与项目数据增加 `user_id/project_id` 维度，查询必须带租户条件。  
5. 增加最小安全回归测试：未登录 401、角色越权 403、用户 A 不可读用户 B 数据。

---

### 问题 2：生产敏感信息硬编码在仓库配置中

**证据**
- `application.yml` 直接包含数据库账号 `root` 和明文密码。 

**影响**
- 凭据泄露风险高；不符合最小暴露原则；发布到公开仓库会直接暴露数据库访问能力。

**解决方案（可执行）**
1. 立刻轮换数据库密码。  
2. 改为环境变量注入：`SPRING_DATASOURCE_URL/USERNAME/PASSWORD`。  
3. 提供 `application-local.yml.example`，仓库不提交真实凭据。  
4. 接入 Secret 管理（K8s Secret / Vault / 云厂商密钥管理）。

---

### 问题 3：前端直接调用 Python 服务固定地址，绕过 Java 业务网关

**证据**
- Dashboard 页通过 `axios.post('http://localhost:8001/api/popular/fetch?pages=5')` 直连 Python。  

**影响**
- 破坏“前端 -> Java -> Python”分层边界；线上环境地址不可移植；鉴权链路断裂（前端可绕过 Java 安全策略）。

**解决方案（可执行）**
1. 前端只调用 Java API。  
2. Java 新增“触发热门抓取”代理接口，服务层调用 Python。  
3. Python 服务收敛为内网服务，不对公网暴露。  
4. 在网关层统一做鉴权、限流、审计日志。

---

## P1（中优先级）

### 问题 4：CORS 配置不安全（`*` + `allow_credentials=true`）

**证据**
- Python FastAPI 中间件设置了 `allow_origins=["*"]` 且 `allow_credentials=True`。  

**影响**
- 违背浏览器 CORS 规范最佳实践；可能引发跨域凭据风险；生产不可控。

**解决方案（可执行）**
1. 区分环境：开发环境白名单 `http://localhost:5173`；生产使用明确域名列表。  
2. 若确需 `credentials=true`，禁止通配符 `*`。  
3. 与 API Gateway 同步跨域策略，避免双重放开。

---

### 问题 5：Maven 构建受仓库访问策略影响，无法稳定构建

**证据**
- `mvn test` 失败：`repo.maven.apache.org` 返回 403，父 POM 无法解析。  

**影响**
- CI/CD 不稳定；新成员无法一键启动；交付风险高。

**解决方案（可执行）**
1. 在项目文档提供 `settings.xml` 镜像模板（如阿里云/腾讯云 Maven 镜像）。  
2. CI 环境预置私有制品代理（Nexus/Artifactory）并锁定镜像。  
3. 增加 `mvn -U -s .mvn/settings.xml verify` 作为标准命令。

---

### 问题 6：实施文档与代码状态不一致，误导开发/测试

**证据**
- 文档称“前端 API 与页面 0%”，但仓库已有相关实现。  
- 文档启动指南引用 `sql/schema_video_review.sql`，实际仓库为 `sql/bili_insight_db.sql`。  

**影响**
- 新成员按文档执行会失败；任务排期和完成度判断失真。

**解决方案（可执行）**
1. 统一文档基线：以当前代码为准重写“已完成/待完成”。  
2. 引入 `docs/CHANGELOG.md` 与版本标签，文档随 PR 强制更新。  
3. 在 CI 增加文档路径存在性检查（引用文件必须存在）。

---

## P2（优化项）

### 问题 7：前端打包体积过大，首屏性能风险高

**证据**
- 构建输出主 bundle 超过 2MB，Vite 发出 chunk size warning。  

**影响**
- 首屏慢、移动端体验差、CDN 成本上升。

**解决方案（可执行）**
1. 路由级与组件级懒加载（已部分使用，继续细分）。  
2. 为 ECharts、Element Plus 做按需引入。  
3. 配置 `manualChunks` 拆分 vendor。  
4. 增加 Lighthouse 性能门禁（例如 LCP/JS 体积阈值）。
